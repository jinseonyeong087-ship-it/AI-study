# 12. 일 시작 전에 알아야 할 내용 (불량검출 직무, 노베이스용 세부 정리)

> 목적: "무엇을 알아야 현업 대화를 따라갈 수 있는지"를 한 문서에서 정리

---

## 0) 이 문서 사용법

- 처음에는 전부 외우려고 하지 말고, **큰 그림 → 핵심 용어 → 실무 포인트** 순서로 읽기
- 각 항목마다 있는 **"현업에서 왜 중요한가"**를 특히 집중해서 보기
- 모르는 용어가 나오면 바로 메모하고, 다음날 다시 확인

## 1) 불량검출 업무의 본질

불량검출은 단순히 "모델 정확도 올리기"가 아니라,
**생산 라인에서 불량을 안정적으로 걸러내는 시스템**을 만드는 일입니다.

핵심은 3가지:
1. **검출 성능** (불량을 잘 찾는가)
2. **운영 안정성** (환경이 바뀌어도 유지되는가)
3. **업무 연계성** (검출 결과가 실제 공정에 도움 되는가)

### 현업에서 왜 중요한가
- 모델 점수만 높아도, 라인에서 오작동하면 쓸 수 없음
- 실제 업무는 "정확도"보다 "재현성과 유지보수"가 더 자주 문제 됨

---

## 2) 반드시 구분할 문제 유형 3가지

## 2-1) 분류(Classification)
- 이미지 1장을 보고 정상/불량을 판정
- 출력: 클래스 1개

**언제 쓰나**
- 제품 단위 판정이 목적일 때

## 2-2) 탐지(Detection)
- 이미지에서 불량의 위치(박스) + 종류를 찾음
- 출력: 박스 좌표 + 클래스 + 신뢰도

**언제 쓰나**
- 불량 위치가 중요하거나, 복수 불량이 있을 때

## 2-3) 분할(Segmentation)
- 불량 영역을 픽셀 단위로 정확히 구분
- 출력: 영역 마스크

**언제 쓰나**
- 면적 계산, 미세 형상 분석이 중요할 때

### 현업 포인트
- 처음에는 탐지(YOLO) 기반이 실무 도입 속도가 빠른 경우가 많음

---

## 3) 데이터가 모델보다 중요하다는 말의 진짜 의미

## 3-1) 데이터 품질 요소
- 라벨 정확성
- 촬영 조건 일관성(조명, 각도, 거리)
- 클래스 균형(정상/불량 비율)
- 실제 현장 다양성 반영

## 3-2) 자주 생기는 데이터 문제
- 불량 샘플이 매우 적음
- 특정 시간대/라인 데이터만 있음
- 라벨 기준이 작업자마다 다름
- 신규 불량 유형이 데이터에 없음

### 현업 포인트
- "모델 바꾸기"보다 "데이터/라벨 정리"가 효과 큰 경우가 압도적으로 많음

---

## 4) 라벨링에서 반드시 알아야 할 것

## 4-1) 라벨링은 정답지 제작
모델은 라벨을 "진실"로 믿고 학습하므로,
라벨 품질이 낮으면 고성능 모델도 실패합니다.

## 4-2) 라벨링 규칙(예시)
- 박스는 불량 영역을 충분히 포함하되 과도하게 크지 않게
- 애매한 케이스는 별도 규칙(제외/포함)을 문서화
- 클래스 이름/ID 매핑은 고정

## 4-3) 라벨 검수
- 랜덤 샘플 수동 검수
- 클래스별 누락 여부 확인
- 동일 이미지 중복 라벨 여부 확인

### 현업 포인트
- 라벨링 가이드 문서가 없는 팀은 성능 편차가 크게 발생

---

## 5) 모델 학습 파이프라인 (반드시 이해)

1. 문제 정의
2. 데이터 수집/정리
3. train/val/test 분리
4. 학습
5. 평가
6. 오류 분석
7. 데이터/라벨/설정 개선 후 재학습

### 중요한 원칙
- 테스트셋은 최종 확인용 (학습 튜닝에 반복 사용 금지)
- 같은 이미지가 train/test에 동시에 있으면 안 됨(데이터 누수)

---

## 6) 성능 지표 해석 (정확도만 보면 안 되는 이유)

## 6-1) 핵심 지표
- Precision(정밀도): 불량이라 한 것 중 실제 불량 비율
- Recall(재현율): 실제 불량 중 찾아낸 비율
- F1-score: precision/recall 균형
- mAP: 탐지 모델 종합 성능 지표

## 6-2) 실무에서 흔한 우선순위
- 불량 놓침 비용이 크면: Recall 우선
- 오검출로 재검 비용이 크면: Precision도 함께 중요

### 현업 포인트
- "우리 공정에서 어떤 실수가 더 치명적인지" 먼저 합의해야 함

---

## 7) 오검출/미검출을 보는 관점

- **오검출(False Positive)**: 정상을 불량으로 판단
  - 영향: 재검 증가, 작업 지연
- **미검출(False Negative)**: 불량을 정상으로 놓침
  - 영향: 불량 유출, 클레임/비용 증가

### 현업 포인트
- 팀마다 허용 가능한 오검출/미검출 비율이 다름
- 수치 목표를 공정팀/품질팀과 함께 맞춰야 함

---

## 8) YOLO를 실무에서 쓸 때 알아야 할 필수 개념

- Bounding Box: 불량 위치 박스
- Confidence: 모델 확신도
- IoU: 예측 박스와 정답 박스 겹침 정도
- NMS: 중복 박스 제거
- mAP50 / mAP50-95: 평가 점수

### 현업 포인트
- confidence threshold를 너무 높이면 미검출 증가
- 너무 낮추면 오검출 증가
- 임계값은 현장 목표 기준으로 조정해야 함

---

## 9) 성능이 갑자기 떨어질 때 점검 순서

1. 카메라/조명/렌즈 상태 변화 여부
2. 제품 포지션/속도/배경 변경 여부
3. 신규 불량 유형 발생 여부
4. 라벨 품질 저하 여부
5. 데이터 분포 변화(계절/시간대/라인 변경)

### 현업 포인트
- 모델 자체보다 "입력 환경 변화"가 원인인 경우가 매우 많음

---

## 10) 전처리/증강에서 알아야 할 원칙

## 10-1) 전처리
- 해상도 통일
- 색상공간/정규화 일관성 유지
- 파일 손상/중복 제거

## 10-2) 증강(Augmentation)
- 밝기 변화, 회전, 노이즈, 블러 등
- 현실에서 가능한 범위 내에서만 적용

### 현업 포인트
- 비현실적 증강은 오히려 성능 악화

---

## 11) 배포/운영 관점에서 꼭 알아야 할 것

- 추론 속도(ms/frame)
- 장비 자원(CPU/GPU/메모리)
- 모델 버전 관리
- 롤백(문제 시 이전 모델 복귀) 전략
- 로그/모니터링 체계

### 현업 포인트
- "학습 성공"과 "운영 성공"은 다름
- 운영에선 안정성과 복구 가능성이 매우 중요

---

## 12) 협업할 때 자주 쓰는 실무 언어

- "이번 버전은 recall 개선, precision 소폭 하락"
- "미검출 케이스는 반사광 조건에 집중됨"
- "라벨 기준 A/B 불일치가 있어 가이드 통일 필요"
- "테스트셋 분포가 실제 라인 분포와 다름"
- "threshold 조정으로 오검출-미검출 트레이드오프 조정 가능"

이런 문장을 이해하고 말할 수 있으면 현업 적응 속도가 빨라집니다.

---

## 13) 입사 전 최소 실전 경험 체크리스트

- [ ] 가상환경 생성/패키지 설치를 스스로 할 수 있다
- [ ] 이미지 폴더를 읽어 기본 통계를 낼 수 있다
- [ ] train/val/test 분리 이유를 설명할 수 있다
- [ ] confusion matrix를 보고 오검출/미검출을 구분할 수 있다
- [ ] YOLO 학습/추론 명령어를 1회 이상 실행해봤다
- [ ] 결과에서 recall/precision/mAP를 읽을 수 있다
- [ ] 오류 케이스를 모아 개선 루프를 설명할 수 있다

---

## 14) 입사 직후 바로 물어보면 좋은 질문

1. 우리 공정에서 더 치명적인 건 오검출인가, 미검출인가?
2. 클래스(불량 유형) 정의와 라벨링 가이드는 어디에 있는가?
3. 현재 성능 기준 지표와 목표 수치는 무엇인가?
4. 데이터 수집/재학습 주기는 어떻게 되는가?
5. 운영 중 성능 저하를 감시하는 지표는 무엇인가?

이 질문을 하면, 팀의 우선순위를 빠르게 파악할 수 있습니다.

---

## 15) 마지막 정리 (진짜 핵심)

1. 불량검출은 모델 싸움이 아니라 **데이터+운영 싸움**이다.
2. 정확도 숫자 하나보다 **오검출/미검출 구조**를 보는 습관이 중요하다.
3. 빠른 반복(실패 분석 → 데이터 보완 → 재학습)이 가장 강력한 실력이다.

이 3가지를 기억하면, 노베이스여도 현업에서 빠르게 성장할 수 있습니다.
