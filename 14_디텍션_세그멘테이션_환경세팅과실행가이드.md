## 4) 디텍션(Detection) 실행

## 4-1) 데이터 구조

```text

  images/
    train/
    val/
  labels/
    train/
    val/
  data.yaml
```

YOLO 라벨 형식:

```text
<class_id> <x_center> <y_center> <width> <height>
```

## 4-2) 학습


## 7) 자주 나는 오류 + 바로 점검

### 오류 A: `yolo: command not found`
- 설치 셀 재실행: `!pip install ultralytics`
- 런타임 재시작 후 다시 실행

### 오류 B: `FileNotFoundError` / data.yaml 경로 오류
- `/content/...` 절대경로로 지정
- zip 해제 위치 확인

### 오류 C: 라벨 관련 에러
- 이미지/라벨 파일명 1:1 대응 확인
- `train/val` 폴더 구조 확인
- 세그 라벨과 디텍션 라벨 형식 혼동 여부 확인

### 오류 D: 너무 느림
- GPU 런타임 사용 가능한지 확인
- `epochs`, `imgsz`를 줄여서 빠른 검증 먼저

## 9) 라벨링 툴별 선택 가이드 (실무 기준)

### A. labelImg
- 용도: 디텍션(사각형 박스) 전용
- 장점: 가볍고 빠름, 입문자 적응 쉬움
- 단점: 세그멘테이션(폴리곤)에는 부적합
- 실무 사용 포인트:
  - 빠른 PoC(개념검증)에서 소량 데이터 라벨링
  - 클래스 1~3개, 박스 검출 중심 작업

### B. labelme
- 용도: 세그멘테이션(폴리곤), 디텍션도 가능
- 장점: 로컬에서 바로 실행, 폴리곤 작업 직관적
- 단점: 기본 출력이 JSON이라 YOLO-seg 학습 전 변환 필요
- 실무 사용 포인트:
  - 소규모/개인 작업에서 빠른 마스크 라벨링
  - 모델 실험 전 초기 데이터셋 제작

### C. CVAT
- 용도: 디텍션/세그멘테이션/트래킹 등 협업 라벨링
- 장점: 팀 협업(검수, 역할 분리), 다양한 포맷 export
- 단점: Docker 기반이라 초기 세팅이 상대적으로 무거움
- 실무 사용 포인트:
  - 라벨러/검수자 분리 운영
  - 데이터 품질 관리(검수 프로세스) 필요 시
  - 프로젝트 장기 운영

### D. Roboflow(웹 기반)
- 용도: 라벨링 + 데이터 버전관리 + 포맷 export
- 장점: 설치 부담 적고 UI 쉬움, YOLO export 편리
- 단점: 업로드/계정 정책 고려 필요(사내 보안 이슈 가능)
- 실무 사용 포인트:
  - 빠르게 시작해야 하는 스타트업/프로토타입
  - 데이터 증강/버전관리까지 한 번에 관리

### 툴 선택 한 줄 기준
- 혼자 빨리 시작: **labelImg/labelme**
- 팀 단위 정식 운영: **CVAT**
- 웹에서 빠른 진행/배포 친화: **Roboflow**
